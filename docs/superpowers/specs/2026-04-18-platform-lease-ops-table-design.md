# 平台运维页租约表格化设计

## 1. 背景

当前平台详情页的“运维”标签页里，“手动绑定账号出口 IP”是一个盲填表单。用户必须自己输入账号名和出口 IP，前端看不到当前平台已有租约，也看不到 `account -> egress_ip` 的现状。

这导致两个问题：

1. 维护动作没有全局视图，容易改错账号。
2. 请求日志页已经有成熟的表格交互，但平台运维页仍停留在表单堆叠，体验不一致。

后端已经具备这次改造所需的基础接口：

- `GET /api/v1/platforms/{id}/leases`
- `POST /api/v1/platforms/{id}/leases/{account}/actions/assign`
- `DELETE /api/v1/platforms/{id}/leases`

所以这次改造聚焦在前端信息组织和交互收口，不新增后端能力。

## 2. 目标

这次改造只解决一件事：把“已有租约的出口 IP 调整”从盲填改成可视化管理。

目标如下：

1. 在平台详情页“运维”标签里展示当前平台已有租约表。
2. 每行提供“编辑”入口，只修改该账号当前租约的 `egress_ip`。
3. 编辑使用居中小面板，不做行内编辑，也不做右侧大抽屉。
4. 保存后刷新租约表和出口 IP 候选数据，并给出明确反馈。
5. 保留现有危险操作区，但和租约管理区分开。

## 3. 非目标

这次不做下面这些内容：

1. 不保留“新增盲填绑定”入口。当前版本只管理已有租约。
2. 不增加筛选、排序、分页 UI。
3. 不新增批量编辑、批量删除、单租约删除。
4. 不改动平台监控面板结构。
5. 不新增后端接口语义，也不改动租约路由热路径。

## 4. 选型结论

采用“租约表格 + 居中小面板编辑”的方案。

理由：

1. 行内编辑虽然快，但会把表格状态、按钮状态和输入状态混在一起，交互容易乱。
2. 右侧抽屉更适合复杂详情。这次只改一个 `egress_ip`，抽屉太重。
3. 居中小面板和当前需求匹配，改动成本低，也能复用现有 `modal-overlay` / `modal-card` 样式。

## 5. 页面结构

“运维”标签页改成上下两段。

### 5.1 上半部分：当前平台租约表

标题建议为“当前平台租约”。

表格复用现有 `DataTable` 和统一表格样式，视觉上与请求日志页保持同一套语言。

表格列定义如下：

1. `账号`
   显示 `account`。
2. `出口 IP`
   显示 `egress_ip`。
3. `节点`
   主信息显示 `node_tag`，次信息显示 `node_hash` 或短标识。
4. `最近访问`
   显示 `last_accessed`，按现有时间格式化规则输出。
5. `到期时间`
   显示 `expiry`，按现有时间格式化规则输出。
6. `操作`
   只放一个 `编辑` 按钮。

表格行为：

1. 不支持整行点击。
2. `操作` 按钮区复用现有 `subscriptions-row-actions` 风格。
3. 数据为空时显示空状态，文案明确为“当前平台还没有活跃租约”。
4. 查询失败时在表格区上方显示错误提示，不影响危险操作区继续可见。

### 5.2 下半部分：危险操作区

保留现有 3 个操作：

1. `重置为默认配置`
2. `清除所有租约`
3. `删除平台`

这些操作继续使用当前卡片式布局，不并入表格。原因是它们语义不同，混在表格里会稀释风险提示。

## 6. 编辑小面板设计

点击某一行的 `编辑` 后，弹出居中小面板。

面板采用现有 `modal-overlay` / `modal-card` 模式，不新造视觉系统。

面板内容如下：

1. 标题：`编辑出口 IP`
2. 说明文案：提示这是覆盖当前账号租约的出口 IP
3. 只读字段：`账号`
4. 可编辑字段：`出口 IP`
5. 底部按钮：`取消`、`保存`

交互规则：

1. 打开面板时，`账号` 取当前行数据，`出口 IP` 预填当前值。
2. `账号` 在面板中只读，不允许改账号。
3. 面板需要展示当前平台可选的出口 IP 候选，候选数据来自 `ip-load` 接口，避免用户改到当前平台不可路由的 IP。
4. 出口 IP 输入控件优先采用“可输入 + 候选提示”模式，而不是纯自由输入。
5. 点击遮罩可关闭。
6. 右上角提供关闭按钮。
7. 支持 `Esc` 关闭。
8. 保存进行中时，`保存` 按钮显示加载状态，避免重复提交。
9. 保存失败时面板不自动关闭，方便用户继续修改。

## 7. 数据流设计

### 7.1 查询

前端新增平台租约查询封装，直接调用现有 `GET /platforms/{id}/leases`。

查询策略：

1. 仅在 `ops` 标签激活时发起。
2. 默认按 `account asc` 拉取，保证管理视图稳定。
3. 本版不做分页 UI，但后端接口仍然是分页式响应。

为了避免静默截断，本版固定请求一个足够大的上限值，例如 `limit=1000`。如果返回结果显示 `total > items.length`，页面需要给出警告，明确当前只展示前 `N` 条租约，而不是假装“全部展示完了”。

### 7.2 出口 IP 候选

前端新增平台出口 IP 负载查询封装，调用现有 `GET /platforms/{id}/ip-load`。

用途只有一个：给编辑小面板提供当前平台可选出口 IP 候选，并尽量让用户从真实可路由的出口里选，而不是盲猜。

候选展示规则：

1. 默认按 `lease_count desc` 拉取，优先让用户看到当前已有负载分布。
2. 候选至少展示 `egress_ip`，可附带 `lease_count` 作为辅助信息。
3. 如果候选查询失败，面板仍可打开，但要明确提示“候选出口加载失败，保存时以后端校验为准”。

### 7.3 保存

保存继续复用现有 `assign lease to egress ip` 接口：

- `POST /platforms/{id}/leases/{account}/actions/assign`

请求体只传：

- `egress_ip`

前端不新增新的写接口，也不引入二次确认。

边界语义需要在实现中保持一致：

1. 用户编辑的是“出口 IP”，不是“具体节点”。
2. 如果同一个出口 IP 对应多个节点，由后端按既有规则选择节点，前端不暴露节点选择。
3. 保存是覆盖写，不是补丁式更新；租约时间字段会按后端当前语义重置。

### 7.4 失效和刷新

保存成功后至少做下面几件事：

1. 关闭编辑面板。
2. 失效当前平台租约表 query。
3. 失效当前平台出口 IP 候选 query。
4. toast 提示“账号 X 已绑定到出口 IP Y”。

清除所有租约成功后，也要失效同一组 query，保证租约表和出口候选一起更新。

监控页签的数据继续遵循现有采样和查询节奏。这次设计不依赖“保存后立刻把监控面板强刷到最新”作为 correctness 条件。

## 8. 校验与错误处理

输入校验维持最小闭环：

1. `出口 IP` 不能为空。
2. 提交前先做 `trim()`。

这次不在前端额外实现复杂 IP 格式校验，理由是后端已经有最终校验和错误返回。前端只负责阻止空值，避免前后端校验规则重复发散。

但有两条后端约束必须明确体现在前端实现里：

1. 提交路径里的 `account` 必须继续走 URL 编码，不能手拼路径。
2. 请求体只能传 `egress_ip`，不能顺手附带其他字段。

错误处理策略：

1. 租约表查询失败：表格区展示错误 callout。
2. 出口 IP 候选查询失败：面板内提示，但不阻止用户继续输入和提交。
3. 保存失败：保留面板，toast 或面板内提示错误。
4. 清除所有租约失败：沿用当前错误提示方式。

## 9. 需要改动的前端边界

主要变更集中在平台模块，不碰请求日志模块。

预期涉及：

1. `webui/src/features/platforms/PlatformDetailPage.tsx`
   增加租约表 query、表格列定义、编辑面板状态和提交逻辑。
2. `webui/src/features/platforms/api.ts`
   增加租约列表和出口 IP 候选查询封装。
3. `webui/src/features/platforms/types.ts`
   按需要补充租约分页输入、IP 负载项或辅助类型。
4. `webui/src/styles/theme.css`
   只补最少量平台运维页专属样式；优先复用已有 `data-table`、`modal-*`、`subscriptions-row-actions`。

## 10. 测试设计

这次以前端回归为主，至少覆盖以下场景：

1. `ops` 标签打开后能看到租约表。
2. 没有租约时空状态正确。
3. 点击 `编辑` 能打开面板，并正确带入账号和当前出口 IP。
4. 面板能加载出口 IP 候选；候选加载失败时也有明确提示。
5. 修改并保存成功后，表格刷新且提示成功。
6. 保存失败时面板不关闭。
7. 清除所有租约后，租约表与出口候选一起刷新。
8. 危险操作区仍能正常工作，没有因为表格改造被挤掉或失效。

## 11. 风险与约束

这次真正的风险不在接口，而在“无分页 UI”与“已有分页接口”之间的边界。

如果某个平台租约量超过前端单次拉取上限，而页面又不提示，就会出现用户以为“表里就是全部租约”，实际只是前一部分。这是必须明确规避的行为风险。

因此本设计要求：

1. 没有分页 UI 可以接受。
2. 但不能静默截断。
3. 一旦超过单次展示上限，必须明确告知用户当前只显示前 `N` 条。

还要明确另外两个后端边界：

1. 目标 `egress_ip` 必须存在于当前平台的可路由视图里，否则保存会返回错误。
2. 同一 `egress_ip` 下如果有多个节点，前端不能假装用户在选节点；真实生效节点由后端按既有规则决定。

## 12. 验收标准

满足下面几点，就算这次设计落地达标：

1. 用户进入平台详情页“运维”标签后，能直接看到当前平台租约表。
2. 用户不再需要手动输入账号名。
3. 用户可通过每行的 `编辑` 按钮修改已有租约的出口 IP。
4. 编辑通过小面板完成，不是行内编辑，也不是右侧抽屉。
5. 保存后表格和出口 IP 候选数据能刷新。
6. `重置为默认配置`、`清除所有租约`、`删除平台` 仍保留并可用。
